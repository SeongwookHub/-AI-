# 네이버 뉴스 키워드 대시보드 — 통합 설계서

> 이 문서는 Claude Code가 구현 시 참조할 설계 문서다. **구현 코드는 포함하지 않는다.**

---

## 1. 작업 컨텍스트

### 1.1 배경 · 목적
사용자가 지정한 키워드를 기준으로 네이버 뉴스를 수집·분류해서 보여주는 로컬 대시보드를 만든다.
사용자가 "업데이트" 버튼을 누를 때마다 네이버 뉴스 검색 API로 최신 기사를 가져와 동기화한다.
대시보드에서 키워드를 자유롭게 추가할 수 있어야 한다.

### 1.2 범위
- 포함: 키워드 CRUD(추가/삭제/조회), 네이버 뉴스 검색 API 연동, 기사 저장·중복 제거, 키워드별 분류 뷰, 수동 업데이트 트리거
- 제외: 자동 스케줄링(cron/데일리 자동 실행) — 사용자가 실행 환경으로 "그때그때 수동 실행"을 선택했으므로 범위에서 제외한다. 추후 필요 시 확장 지점으로만 남겨둔다.
- 제외: 기사 본문 요약/감성분석 등 LLM 기반 콘텐츠 가공 — 분류 방식이 "단순 키워드 포함 매칭"으로 확정되었으므로 런타임에 LLM 호출이 필요 없다.

### 1.3 입력 · 출력 정의

| 구분 | 내용 |
|---|---|
| 입력 | (1) 사용자가 등록한 키워드 목록 (2) 네이버 뉴스 검색 API 응답(JSON: title, link, description, pubDate, originallink 등) |
| 출력 | (1) 로컬에서 열람 가능한 대시보드 UI (2) 키워드별로 분류된 기사 리스트(최신순) (3) 저장된 기사 데이터(파일 또는 DB) |

### 1.4 주요 제약조건
- 네이버 오픈API는 **Client ID/Secret**이 필요하며, 이 값은 코드에 하드코딩하지 않고 `.env` 등 별도 설정 파일로 분리하고 `.gitignore` 처리한다.
- 네이버 뉴스 검색 API는 호출당 최대 100건, 요청 빈도 제한(초당 호출 수 제한)이 있으므로 키워드 수가 많아지면 순차 호출 + 짧은 딜레이가 필요하다.
- 실행 환경이 "수동 실행"이므로 백그라운드 프로세스나 스케줄러 설정 없이, 사용자가 대시보드를 열고 버튼을 누르는 시점에만 외부 API 호출이 발생해야 한다.
- 로컬 환경 실행을 전제로 하므로 배포용 인프라(서버, 컨테이너 오케스트레이션)는 설계에서 제외한다.

### 1.5 용어 정의

| 용어 | 정의 |
|---|---|
| 키워드 | 사용자가 등록한 뉴스 분류 기준 문자열 (예: "반도체") |
| 동기화(sync) | 업데이트 버튼 클릭 시 네이버 API를 호출해 신규 기사를 가져와 저장하는 동작 |
| 매칭 | 기사의 제목/설명에 키워드 문자열이 포함되는지 확인하는 단순 문자열 포함 검사 |
| 게이트 | 특정 산출물이 생성/수정될 때 자동 실행되는 검증 스크립트(훅) |

## 2. 워크플로우 정의

### 2.1 전체 흐름

```
[대시보드 진입]
      │
      ├─▶ (A) 키워드 관리 흐름
      │      사용자가 키워드 입력 → 형식 검증(코드) → keywords.json에 추가
      │      → 대시보드 키워드 목록 갱신
      │
      └─▶ (B) 업데이트(동기화) 흐름
             [업데이트 버튼 클릭]
                   │
                   ▼
             키워드 목록 로드 (코드)
                   │
                   ▼
             키워드별 네이버 뉴스 검색 API 순차 호출 (코드)
                   │
                   ▼
             응답 파싱 → 구조화 JSON 생성 (코드)
                   │
                   ▼
             스키마 검증 (코드/게이트) ──FAIL──▶ 해당 키워드 결과 스킵 + 에러 로그, 나머지 계속 진행
                   │ PASS
                   ▼
             중복 제거 (link 기준, 코드)
                   │
                   ▼
             키워드 매칭 태깅 (단순 포함 매칭, 코드)
                   │
                   ▼
             저장소에 upsert (코드, 멱등 규칙 적용)
                   │
                   ▼
             대시보드 재렌더링 (키워드별 최신순 표시)
```

### 2.2 단계별 상세

| 단계 | 유형 | 성공 기준 | 검증 방법 | 실패 처리 |
|---|---|---|---|---|
| 키워드 추가 | 코드 | 중복 없는 문자열, 빈 값 아님 | 입력 검증 스크립트 | 에러 메시지 반환, 저장 거부 |
| API 호출 | 코드 | HTTP 200, JSON 파싱 성공 | 응답 코드 체크 | 재시도(최대 N회) 후 실패 시 해당 키워드만 스킵 + 로그 |
| 구조화 JSON 생성 | 코드 | 필수 필드(title, link, pubDate) 존재 | 스키마 검증 스크립트 | 필드 누락 항목 제외, 나머지 저장 |
| 중복 제거 | 코드 | link 기준 유일성 확보 | dedup 스크립트 | 중복 항목 자동 스킵 |
| 키워드 매칭 | 코드 | 제목/설명에 키워드 문자열 포함 여부 정확히 판정 | 단위 테스트(문자열 포함 로직) | 매칭 실패 시 "미분류"로 태깅, 로그 |
| 저장(upsert) | 코드 | link 키 기준 덮어쓰기, 없으면 삽입 | 저장 후 read-back 비교 | 실패 시 해당 배치 롤백 |
| 대시보드 렌더링 | 코드 | 키워드별 섹션에 기사 정상 표시 | UI 스모크 테스트 | 렌더링 실패 시 에러 배너 표시 |

> 참고: 이 시스템은 판단(LLM) 영역이 거의 없다. 키워드 매칭이 단순 포함 매칭으로 확정되었기 때문에, 워크플로우 전체가 결정론적 스크립트로 처리 가능하다. LLM은 **구현 단계(Claude Code가 코드를 작성/검토하는 과정)**에서만 사용되고, **런타임 동작에는 관여하지 않는다.**

### 2.3 분기 조건
- API 호출 실패(네트워크/인증 오류) → 해당 키워드만 건너뛰고 나머지 키워드 계속 처리, 최종 결과에 "동기화 실패 키워드 목록" 표시
- 신규 키워드 추가 시 기존 저장된 기사에 대해서도 재매칭할지 여부 → **기본값: 재매칭하지 않음**(추가 시점 이후 수집분부터 적용). 과거 데이터 소급 매칭이 필요하면 별도 "전체 재매칭" 버튼을 향후 확장 지점으로 남겨둔다.

## 3. 구현 스펙 (구조 · 역할 수준)

### 3.1 폴더 구조

```
/naver-news-dashboard
  ├── CLAUDE.md
  ├── .env                          # NAVER_CLIENT_ID, NAVER_CLIENT_SECRET (gitignore)
  ├── /.claude
  │   ├── settings.json             # 훅(자동 게이트) 정의
  │   └── /skills
  │       ├── /naver-api-client/    # SKILL.md + /scripts + /references
  │       ├── /news-matcher/        # SKILL.md + /scripts
  │       └── /schema-validator/    # SKILL.md + /scripts
  ├── /backend
  │   ├── /routes                   # 키워드 CRUD, 동기화 트리거 엔드포인트
  │   ├── /services                 # 네이버 API 연동, 매칭, dedup 로직
  │   └── /models                   # 기사/키워드 데이터 모델
  ├── /frontend
  │   └── (대시보드 UI: 키워드 관리 화면 + 업데이트 버튼 + 키워드별 기사 리스트)
  ├── /output
  │   ├── keywords.json             # 키워드 목록
  │   ├── articles.db (or .json)    # 수집된 기사 저장소
  │   └── _pipeline_state.json      # 단계 통과 기록(스크립트만 기록, 직접 편집 금지)
  └── /docs
```

> 서브에이전트 폴더(`/.claude/agents`)는 두지 않는다 — 아래 3.3 참고.

### 3.2 기술 스택 제안 (가정 · 구현 시 조정 가능)
"수동 실행 + 로컬 환경"이라는 조건에 맞춰 가벼운 조합을 기본값으로 제안한다:
- 백엔드: Python (FastAPI 또는 Flask) — 네이버 API 호출, 매칭/저장 로직
- 저장소: SQLite 또는 단순 JSON 파일 — 로컬 단일 사용자 환경이므로 별도 DB 서버 불필요
- 프론트엔드: 단일 페이지(Vanilla JS/HTML 또는 경량 프레임워크) — 키워드 관리 폼 + 업데이트 버튼 + 키워드별 기사 카드 리스트
- 실행: `python app.py` 등으로 로컬 서버 기동 → 브라우저에서 `localhost` 접속 (스케줄러/배포 설정 없음)

### 3.3 에이전트 구조: **단일 에이전트**
- 워크플로우가 단순하고(키워드 CRUD + API 연동 + 저장 + 렌더링), 판단 영역이 거의 없어 지침이 길지 않다.
- 서브에이전트로 분리할 만큼 독립적인 도메인 지식이나 상호 오염 위험이 없으므로 **CLAUDE.md 하나 + 스킬 조합**으로 처리한다.
- 향후 기능이 커지면(예: 기사 요약, 감성 분석 추가) 그 시점에 별도 서브에이전트 분리를 재검토한다.

### 3.4 단계별 처리 방식 (판단 vs 스크립트)

| 단계 | 처리 방식 |
|---|---|
| 키워드 입력 검증 | 스크립트 |
| 네이버 API 호출 | 스크립트 |
| 응답 → 구조화 JSON | 스크립트 |
| 스키마 검증 | 스크립트(게이트) |
| 중복 제거 | 스크립트 |
| 키워드 매칭(포함 매칭) | 스크립트 |
| 저장(upsert) | 스크립트 |
| 대시보드 렌더링 | 스크립트 |
| (구현 단계 한정) 코드 리뷰·엣지케이스 판단 | Claude Code 판단 영역 |

### 3.5 CLAUDE.md 핵심 섹션 목록
- 프로젝트 개요 및 목적
- 폴더 구조 안내
- 네이버 API 사용 규칙(요청 제한, 인증 방식, `.env` 참조 방법)
- 데이터 스키마 정의(기사, 키워드)
- 훅/게이트 동작 방식 요약 및 `_pipeline_state.json` 직접 수정 금지 안내
- 멱등 규칙(재실행 시 link 기준 upsert)
- 테스트/검증 실행 방법

### 3.6 스킬 목록 · 역할 · 트리거 조건

| 스킬 | 역할 | 트리거 조건 |
|---|---|---|
| `naver-api-client` | 네이버 뉴스 검색 API 호출 래퍼, 요청 제한 처리, 응답 파싱 | 동기화(업데이트) 요청 시 |
| `news-matcher` | 기사-키워드 포함 매칭 로직 | 신규 기사 저장 전 |
| `schema-validator` | 기사/키워드 JSON 스키마 검증 | API 응답 파싱 직후, 저장 직전 |

### 3.7 서브에이전트
없음 (3.3 참고). 향후 분리 필요 시 `AGENT.md`에 역할·트리거·입출력·참조 스킬을 명시하는 구조를 그대로 적용한다.

### 3.8 데이터 전달 방식
- 키워드 목록: 파일 기반 (`/output/keywords.json`) — 여러 단계에서 재사용되므로
- API 응답 → 매칭 결과: 파일 기반 (`/output/articles.db` 또는 JSON) — 구조화 데이터이며 누적 저장 필요
- 업데이트 버튼 클릭 → 백엔드 트리거: 프롬프트/요청 인라인(단순 액션 호출)

### 3.9 산출물 파일 형식
- `keywords.json`: `[{ "keyword": string, "created_at": string }]`
- 기사 저장소 스키마: `{ "link": string(PK), "title": string, "description": string, "pubDate": string, "keywords": string[], "synced_at": string }`
- `_pipeline_state.json`: `{ "<step-name>": { "status": "pass"|"fail", "hash": string, "timestamp": string } }`

## 4. 하네스 스펙

### 4.1 자동 게이트(훅)

| 트리거 파일 | 검사 스크립트 | PASS 시 | FAIL 시 |
|---|---|---|---|
| `keywords.json` 수정 | `validate_keywords.py` (중복/빈 값/형식 검사) | 대시보드 키워드 목록 갱신 허용 | 변경 거부, 에러 메시지 반환 |
| API 원시 응답 저장 (`/output/_raw/*.json`) | `validate_article_schema.py` (필수 필드 존재 확인) | 매칭 단계로 체이닝 | 해당 항목 제외 + 로그 기록, 나머지 계속 진행 |
| 매칭 결과 저장 (`/output/articles.*`) | `validate_dedup.py` (link 유일성 확인) | 저장 확정 | 중복 항목 자동 제거 후 재검증 |
| 백엔드 코드 변경 (`/backend/**`) | `run_tests.sh` (단위 테스트 실행) | 커밋/병합 허용 | 테스트 실패 목록 반환, 병합 차단 |

### 4.2 파이프라인 상태 파일
- 위치: `/output/_pipeline_state.json`
- 기록 항목: `keyword-validated`, `api-fetched`, `schema-validated`, `deduped`, `matched`, `saved` — 각 단계별 통과 여부 + 처리한 파일/데이터의 해시 + 타임스탬프
- 스크립트만 기록하며, 사람이나 에이전트가 직접 편집 금지
- 다음 단계 실행 거부 조건: 선행 단계 기록이 없거나(`status != pass`), 입력 파일 해시가 마지막 기록과 다를 경우(= 선행 단계가 재검증되지 않은 상태에서 스킵된 것으로 간주) 실행을 거부하고 선행 단계 재실행을 요구

### 4.3 Spec-First 적용 단계
- **적용 지점**: 네이버 API 응답 → 대시보드 표시로 이어지는 데이터 흐름
- 순서: `API 원시 응답 → 구조화 JSON(스키마: link/title/description/pubDate/keywords) → 코드 스키마 검증 → 대시보드 UI 렌더링(자연어/HTML 표시)`
- 검증 전 단계에서 임의로 요약하거나 필드를 가공한 자유 서술 형태로 저장하는 것을 금지한다 — 항상 구조화 JSON을 거친 뒤에만 화면에 표시한다.

### 4.4 멱등 규칙
- 기준 키: 기사의 `link` (원문 URL)
- 재실행(업데이트 버튼 재클릭) 시 동일 `link`를 가진 기사는 기존 레코드를 덮어쓰기(upsert)한다. `keywords` 필드는 기존 태그와 신규 매칭 태그를 합집합(dedup)으로 병합한다.
- 신규 키워드 추가 후 재동기화 시, 과거에 이미 저장된 기사에 대한 소급 재매칭은 하지 않는다(2.3 분기 조건 참고).

### 4.5 환각 차단
- 이 시스템은 런타임에 LLM이 기사 내용을 요약·재구성하지 않고, 네이버 API가 반환한 필드(title, description, link, pubDate)를 그대로 저장·표시하므로 환각 위험 자체가 낮다.
- 다만 향후 기능 확장(예: 기사 요약, 키워드 트렌드 코멘트 등 LLM 생성 텍스트 추가) 시를 대비해 규칙을 미리 명시한다: **LLM이 생성하는 모든 인용·근거 문구는 저장된 기사 원문 필드(title/description)의 부분문자열이어야 하며, 검증 스크립트가 이를 확인하지 못하면 FAIL 처리한다.**

### 4.6 워크트리 격리 대상
- 네이버 API 연동 모듈(`naver-api-client` 스킬) 변경/리팩토링
- 저장소 스키마 변경(마이그레이션 수반 작업)
- 대시보드 UI 대규모 개편
- 위 작업은 `git worktree add ../실험 -b 실험브랜치`로 별도 워크트리에서 진행하고, 검증 통과 후 병합, 실패 시 `git worktree remove`로 제거한다.

## 5. 확인이 필요한 가정 (리뷰 시 확정 요망)

1. 저장소를 SQLite로 할지 JSON 파일로 할지 — 키워드/기사 수가 적다면 JSON으로도 충분
2. 프론트엔드 프레임워크 선택(Vanilla JS vs React 등) — 기능이 단순해 Vanilla로 시작 후 필요 시 확장 제안
3. "과거 기사 소급 재매칭" 기능을 지금 범위에 포함할지, 향후 확장으로 미룰지
4. 네이버 API 요청 제한(초당 호출 수)에 따른 키워드 수 상한 여부
